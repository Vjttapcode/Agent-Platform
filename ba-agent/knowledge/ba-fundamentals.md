# Business Analysis Fundamentals

## Requirement types
- **Business requirements** — the *why*: goals, objectives, success metrics.
- **Stakeholder requirements** — needs of specific user groups.
- **Functional requirements (FR)** — *what* the system must do.
- **Non-functional requirements (NFR)** — *how well*: performance, security,
  availability, usability, scalability, compliance, accessibility.
- **Constraints** — fixed limits: budget, tech stack, deadlines, regulations.

## INVEST — qualities of a good user story
- **I**ndependent — minimal overlap with other stories.
- **N**egotiable — a conversation starter, not a contract.
- **V**aluable — delivers value to a user or buyer.
- **E**stimable — the team can size it.
- **S**mall — fits within an iteration.
- **T**estable — has clear acceptance criteria.

## Good acceptance criteria
- Testable and unambiguous.
- Cover happy path, alternate paths, and error/edge cases.
- Independent of implementation details.
- Gherkin (Given/When/Then) is the default form.

## MoSCoW prioritization
**Must / Should / Could / Won't (this time)** — use to scope a release.
