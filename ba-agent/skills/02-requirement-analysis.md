# Skill: Requirement Analysis

**When:** the user describes a feature/product and wants it analyzed, or asks to
"analyze requirements".

**Produce (in this order):**

1. **Summary** — 2–3 sentences restating the goal.
2. **Stakeholders** — table: role · interest · influence.
3. **Functional requirements** — table: `FR-###` · description · priority (MoSCoW).
4. **Non-functional requirements** — table: `NFR-###` · category · target.
5. **Assumptions** — explicit, labelled.
6. **Constraints & dependencies**.
7. **Risks** — table: risk · impact · likelihood · mitigation.
8. **Open questions** — `Q-###` list of anything blocking confidence.

Be exhaustive on FRs but avoid duplicates. Tie NFRs to concrete, testable
targets (e.g. "p95 latency < 300ms") rather than vague adjectives.
